# Runtime leak audit — 2026-06-10

_Three-lens adversarial workflow (7 agents, ~273 tool calls) over Coro park/resume, analysis preemption, detours, bus subscriptions, interp/env lifetimes, world stores, fd/conn lifecycle._

## Executive summary

The runtime does NOT leak goroutines, interpreters, or fds — the coro lifecycle is sound (turnCancel always registered+cancelled, every interpreter wait/loop/suspend point is ctx-aware, executeWithDetours always joins <-coro.Done, RunHost/registry teardown is complete across x100 restart cycles). What it DOES leak, verified by three adversarial passes: (1) one dead '*' event-bus subscriber channel per interpreter construction — every turn, every detour, every /script, every ANALYSIS command — because startEventTranslator never Unsubscribes; only host restart (bus.Close) frees them. Overnight this is 600-9,000 dead channels/host pinning 5-100MB and adding O(dead-subs) select work to every Publish ON THE FRAME PUMP goroutine. (2) Append-only JSONL (debughttp full-event firehose + decisions.jsonl) with zero rotation — tens of GB across a 200-drone night. (3) Host.Follow leaks a 256-buffer subscriber per call. Separately, two deterministic CORRECTNESS bugs live at exactly the seam Alex flagged: the host-global h.routineCtx is left pointing at the detour's cancelled ctx after every park/resume (poisoning all namespace verbs in the resumed grind — fatigueArbiter makes this fire on every nap), and Pause/ANALYSIS does not cancel a running detour (a 'frozen' host keeps autonomously acting for up to 3 minutes while the operator issues commands). Alex's canonical orphaning scenario is REFUTED — the original routine is reliably terminated and joined. Three one-line patches kill the dominant leaks and are safe to land before tonight; the bus SubscriberCount gauge + pprof + store-size gauges make the soak measure instead of guess.

## Confirmed findings (ranked)

### 1. [CRITICAL] startEventTranslator never Unsubscribes: one dead '*' bus subscriber channel leaked per interpreter construction (every turn, detour, /script, ANALYSIS command) — the dominant leak

**Evidence:** runtime/dsl_events.go:41 `sub := h.bus.Subscribe("*", 64)`; goroutine exits on ctx.Done/bus-close (dsl_events.go:45-50) but never Unsubscribes — the ONLY Unsubscribe call sites in the entire non-test tree are debughttp/server.go:120 and :251 (grep-proven, re-verified). event/bus.go:31-37 retains the chan until Unsubscribe (:46-59) or Close (:90-103); Publish select-tries every dead chan per event under RLock (:67-86) on the FRAME-PUMP goroutine (host.go:614). Built per interpreter at dsl_bridge.go:248, reached from: every conductor turn BOTH paths (conductor.go:452/454 — fixed-routine load drones included; coro path detour.go:84 -> coro.go:34), every detour (detour.go:169), every debughttp POST /script (server.go:372), every ANALYSIS command (analysis.go:296), NewREPL (repl.go:44). Each dead chan pins up to 64 events (AgentThought payloads 1-4KB). Overnight: 600-9,000 dead chans/host = 5-100MB/host + O(leaked) publish CPU per event; freed only at host restart, so the healthiest long-lived host shows the worst slope. Goroutine count stays FLAT — the translator exits; the leak is the bus slice + buffered chans, so only a bus gauge catches it.

**Fix:** (a) `defer h.bus.Unsubscribe("*", sub)` as the first statement inside the goroutine at dsl_events.go:42 (Unsubscribe is mutex-guarded and a safe no-op after Close, bus.go:49-51). (b) Bound translator lifetime to the RUN: `rctx, cancel := context.WithCancel(ctx); defer cancel()` at the top of RunRoutine/RunRoutineSource (dsl_bridge.go:303/316) and inside StartCoro's goroutine before close(c.done) (coro.go:32-52) — this also kills leak #5. (c) Move ParseRoutineString before NewRoutineInterpreter in StartCoro (coro.go:34 vs 43-45) so a parse failure doesn't subscribe. Do NOT touch debughttp Activate (server.go:181) — its translator must live for the session.

### 2. [HIGH] Append-only JSONL never rotated: debughttp full-event firehose + decisions.jsonl — unbounded disk, same path re-appended across restart cycles

**Evidence:** debughttp/server.go:101-111 opens O_APPEND at /tmp/cradle_debug/<user>_events.jsonl; record() (server.go:144-167) appends EVERY bus event (per-tick positions, NPCs, ground items) with no size cap or rotation; cradle/api.go:66-68 and runhost_bootstrap.go:141-149 reopen the SAME file each restart cycle. runtime/decisions.go:41 decisions.jsonl likewise O_APPEND, unbounded. fd lifecycle itself is clean (closeOnce server.go:136-142, defer at :119-120; decisions.go:46). Rate: 10-30 ev/s x 300-600B/line = 3-50MB/hour/host -> 0.1-1GB/host overnight, x200 drones = tens-to-hundreds of GB on /tmp in one soak night.

**Fix:** For tonight: add a Config kill-switch (ring-only, JSONL off) or a kind-allowlist (agent_thought, system_message, policy_veto) for the soak fleet — S. Proper fix: byte-counting rotation in record() (rotate at 64-256MB, keep 1-2 generations) applied to both files, plus export current file size as a gauge.

### 3. [HIGH] h.routineCtx host-global: every detour park/resume DETERMINISTICALLY poisons the resumed grind's namespace verbs with a cancelled ctx; debughttp /script + Activate are genuine data races against it

**Evidence:** dsl_bridge.go:253 `h.routineCtx = ctx` — plain field (host.go:364-370, no mutex), written by whatever goroutine constructs an interpreter; read at EVERY namespace-verb access via boundAction (namespace_actions.go:36,49-60). Flat builtins are immune (ctx captured at construction, dsl_bridge.go:290). Deterministic contamination: runDetour's deferred dcancel (detour.go:167-168) cancels dctx BEFORE executeWithDetours Resumes the grind (detour.go:116); the detour's interpreter left routineCtx=dctx, so every bank.*/trade.*/shop.*/magic.*/prayer.*/combat.* call in the resumed grind binds a CANCELLED ctx until the next turn — fails instantly with context.Canceled. fatigueArbiter makes park/resume routine, so this fires on every nap. Genuine unsynchronized races (go-race detectable): debughttp POST /script (server.go:372) and Activate (server.go:181) write routineCtx from HTTP goroutines while a live conductor routine reads it; plus the runDetour ctx.Done-abandon window. ANALYSIS variant: a cradle HTTP directive leaves routineCtx = the finished request's cancelled ctx, breaking namespace verbs in the persistent /eval session until autonomy resumes. Memory impact trivial; this is the live correctness bug that makes detours destroy the grind they exist to preserve.

**Fix:** Eliminate the host-global: the per-interpreter views are already built in NewRoutineInterpreter (dsl_bridge.go:257-266) — carry ctx on the views (or a per-interpreter binding struct) and have namespaceAction/boundAction read it from the view, making namespace verbs identical to flat builtins. Interim (smaller): save/restore routineCtx around the detour in runDetour under a mutex. Either way add a -race CI run over conductor+debughttp tests.

### 4. [HIGH] Pause/ANALYSIS does not cancel a running detour: a 'frozen' host keeps autonomously acting for up to 3 minutes, concurrent with operator commands

**Evidence:** detour.go:167 `dctx, dcancel := context.WithTimeout(ctx, timeout)` — parented to the CONDUCTOR ctx, not turnCtx (re-verified in source). Pause (conductor.go:256-264) cancels only c.turnCancel = the GRIND turn's cancel; nothing registers dcancel. runCommand (analysis.go:294-296) has no gate on conductor/detour state. fatigueDetourTimeout=3m (detour.go:46-49) and fatigueArbiter makes sleep detours routine. Deterministic whenever ANALYSIS enters mid-detour: Pause kills the PARKED grind (suspend.go:81-84 ctx escape) but the detour keeps walking/sleeping; the operator, believing the host frozen, issues commands -> two interpreters drive one Host (movement contention + the #3 routineCtx clobber). Bounded by the 30s/3m timeout — an execution leak into the freeze window, not a memory leak.

**Fix:** While a detour runs, swap c.turnCancel (under pauseMu) for a func cancelling BOTH turnCtx and dctx, restoring on detour exit — so Pause freezes promptly even mid-nap. CAUTION: do NOT instead re-parent dctx to turnCtx — turnCtx's 2m deadline keeps ticking while the grind is parked and would kill 3m fatigue naps; only do that if you simultaneously extend/suspend the turn deadline during detours.

### 5. [MEDIUM] In-game ANALYSIS operator COMMANDS leak one LIVE translator goroutine + fully pinned interpreter graph each (host-lifetime ctx flows into RunRoutineSource)

**Evidence:** Ctx chain verified: runhost_bootstrap.go:47 per-host ctx -> :115 socialReflex -> reflexAnalysis :352 -> host.Analyze(ctx,...) at :447/:457 -> analysis.go:198/233 runCommand -> :296 RunRoutineSource(host-lifetime ctx) -> dsl_bridge.go:324/248 startEventTranslator(ctx). The translator goroutine lives until host stop, pinning the whole interpreter graph (~200+ actionCallables capturing host+ctx, 10 views, last run's handler maps) AND actively forwarding every bus event into the never-drained 64-cap it.Events (fills, then per-event drop+Debug log forever, dsl_events.go:56-65). Only COMMAND-classified directives build interpreters (ask/hypothetical do not). The cradle HTTP directive path is goroutine-clean (r.Context(), cradle/api.go:168) — it leaks only the #1 channel.

**Fix:** Covered by fix 1(b) (per-run WithCancel inside RunRoutineSource/RunRoutine/StartCoro). Independently worthwhile: bound operator commands in runCommand (analysis.go:294) with a 2-minute WithTimeout + defer cancel.

### 6. [MEDIUM] runDetour: no dctx.Done select case and no join on the ctx.Done arm — bounded today, ONE future non-ctx-aware action handler away from a permanent conductor wedge

**Evidence:** detour.go:170-174 `select { case <-d.Done(): case <-ctx.Done(): }` (re-verified in source — no dctx.Done case, no join on the ctx.Done arm). The 30s/3m timeout is enforced ONLY transitively: dctx deadline -> interpreter per-statement ctx checks unwind the detour -> close(done). Today every blocking handler is ctx-aware (verified: zero raw time.Sleep in non-test runtime code), so nothing is reachable — but one future handler that blocks without selecting on ctx means a timed-out detour never closes done, ctx stays healthy, and runDetour blocks FOREVER: conductor wedged mid-turn, grind parked for host lifetime, host looks alive (frame pump still running). Also: on conductor-ctx.Done (host stop) runDetour returns while the detour coro is live and the grind gets Resumed — a <1s two-dying-routines overlap, harmless today.

**Fix:** Add `case <-dctx.Done():` followed by a bounded join: `select { case <-d.Done(): case <-time.After(5*time.Second): c.log.Error("detour coro failed to exit after cancel — leaked") }` so a stuck detour is detected and the grind is never resumed concurrently with a live detour. (Abort path detour.go:104-105 already joins correctly.)

### 7. [LOW] Host.Follow leaks one 256-buffer '*' subscriber per call (DSL-exposed builtin); bonus: no success-return, so follow() consumes the rest of its turn

**Evidence:** runtime/follow.go:30 `sub := h.bus.Subscribe("*", 256)` (re-verified in source) with NO Unsubscribe on any return path (:40 startup timeout, :43 bus closed, :59 send error, :69 ctx done). Exposed as the `follow` DSL action (dsl_actions.go:54, actions_ambient.go:191-204) — a mesa-authored routine looping follow() on a never-visible target leaks per attempt. Each leak = 1 chan + up to 256 pinned events (~10-100KB). Side bug: after the follow packet is accepted, the loop (follow.go:66-91) has no success exit — only ctx.Done — so every successful follow blocks until turn-ctx death and reports a ctx error.

**Fix:** `defer h.bus.Unsubscribe("*", sub)` immediately after follow.go:30. Separately (behavioral, not tonight): give Follow a bounded duration or success return.

### 8. [LOW] Trust ledger (limbic.Ledger) has no entry cap and is the one store pruneCognition forgot — unbounded per-distinct-player-name growth, whole blob re-serialized to bbolt + mesa every 30s

**Evidence:** runtime/host.go:474 limbic.NewLedger() — no cap; ledger.go has value squash() but no entry eviction (no Prune/maxEntries; Export walks all rows). pruneCognition (runtime/limbic.go:124-142) bounds knowledge (512 subjects) + goalGraph (64 nodes) ONLY. Growth driver: ledger.Met on every ChatReceived/OtherPlayerChat/PrivateMessage/TradeRequest/DuelRequest (limbic.go:199-230); flushLimbic (:351-378) marshals the FULL row set to bbolt AND mesa every 30s; loadLimbic restores it across restarts. Tonight's drone fleet: ≤~200 entries/host, harmless. Long-lived host on a populated server: thousands of entries over weeks with per-flush I/O growing linearly. Verdict: leak-likely (slow, real, unbounded).

**Fix:** Add Ledger.Prune(maxEntries=512) evicting lowest-(Familiar, recency) rows, called from pruneCognition alongside knowledge/goalGraph. ledger.relationships is already a mesa telemetry metric (telemetry.go:58) — watch it tonight.

### 9. [LOW] Teardown race: bus.Subscribe after bus.Close panics on the nil subscribers map — reachable via /ws or /script in the restart window between bus.Close and onExit's a.live delete

**Evidence:** event/bus.go:34 appends into b.subscribers which Close nils (:102); a GET /ws or POST /script routed to the old debughttp handler in the window between bus.Close (inside r.run's RunHost defers) and cradle onExit's a.live delete (api.go:75-79) calls Subscribe on the nil map -> panic. net/http recovers it per-request, so it's a poisoned response + log noise per restart race, not a crash — but it fires on every restart that overlaps a poll.

**Fix:** Guard Subscribe with the closed flag: when b.closed, return a pre-closed channel (callers already handle `!ok`).
